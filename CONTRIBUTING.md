# Contributing to ENGIOS

Thank you for your interest in ENGIOS. This document explains how contributions work and what to expect.

---

## Project Structure

ENGIOS is currently a **solo project.** The founder makes all final decisions regarding what enters the codebase, the architecture, and the direction of the project.

This is not a barrier to participation ‚Äî it is a quality control measure. ENGIOS has a precise philosophy and a specific vision. Maintaining that coherence during early development requires a single point of authority.

This may evolve as the project matures.

---

## How You Can Contribute

### Ideas and Suggestions
Open an **Issue** and label it `idea` or `suggestion`. Describe your thinking clearly. All ideas are read and considered. Not all will be implemented ‚Äî but all are genuinely appreciated.

### Bug Reports
Open an **Issue** and label it `bug`. Include:
- What you expected to happen
- What actually happened
- Your hardware and environment details
- Steps to reproduce

### Code Contributions
1. **Open an Issue first** ‚Äî describe what you want to build and why. Wait for a response before writing code. This prevents wasted effort on contributions that may not align with the project direction.
2. **Fork the repository**
3. **Create a branch** ‚Äî name it clearly: `feature/your-feature-name` or `fix/issue-description`
4. **Write clean, documented code** ‚Äî code that enters ENGIOS should be readable, purposeful, and consistent with the existing codebase
5. **Submit a Pull Request** ‚Äî reference the original Issue, explain your changes clearly

All Pull Requests are reviewed by the founder. Acceptance is not guaranteed. A declined PR is not a rejection of you ‚Äî it is a decision about project direction.

### Discussions
Use **GitHub Discussions** for broader conversations about direction, philosophy, architecture, and ideas that are not yet specific enough for an Issue.

---

## What We Are Looking For

Contributions that align with the ENGIOS philosophy:

- **Human-first design** ‚Äî does this serve the user or extract from them?
- **Privacy by default** ‚Äî does this keep data local?
- **Simplicity** ‚Äî does this make things simpler or more complex?
- **Honesty** ‚Äî is this transparent about what it does?

Contributions that contradict these principles will not be accepted regardless of technical quality.

---

## Code Style

Consistent style guidelines will be published as the codebase matures. Until then ‚Äî write clean, readable, well-commented code and follow the patterns already established in the repository.

## A Note on Direction

ENGIOS has a vision. It is documented. It is deliberate. Contributions that push the project in a fundamentally different direction ‚Äî however well-intentioned ‚Äî will not be accepted.

If you have a different vision for what an OS should be ‚Äî fork the project. That is exactly what open source is for. Build your version. The licence permits it and the spirit of open source encourages it.
